# MCP를 도입하는 이유

## REST API와 비교

기존에 연구소의 서비스를 PO 플랫폼에 연동하려면 이런 방식이었다.

```
PO 플랫폼 코드
  ├── PDF 변환 서비스 → POST /api/pdf/convert 호출 (직접 통합)
  ├── AI 업스케일 서비스 → POST /ai/upscale 호출 (직접 통합)
  ├── HWP 변환 서비스 → POST /hwp/to-pdf 호출 (직접 통합)
  └── ... 새 서비스마다 통합 코드 추가 필요
```

서비스가 늘어날수록 PO 플랫폼 코드도 계속 늘어나고,  
엔드포인트 변경이나 파라미터 변경이 생기면 PO쪽도 같이 수정해야 한다.

---

## MCP 방식

MCP(Model Context Protocol)는 모든 서비스가 동일한 인터페이스를 갖도록 표준화한다.

```
PO 오케스트레이터
  ├── 연구소 MCP 서버 등록 (URL 하나)
  └── 이후 코드 변경 없음

연구소 MCP 서버
  ├── tools/list → 제공 가능한 툴 목록 자동 노출
  └── tools/call → 실제 서비스 실행
```

연구소가 새 서비스를 추가해도 PO 오케스트레이터는 **코드를 한 줄도 바꾸지 않는다.**  
오케스트레이터가 `tools/list`를 다시 수집하면 새 툴이 자동으로 반영된다.

---

## 핵심 장점 요약

| 항목 | REST API 직접 연동 | MCP |
|------|-------------------|-----|
| 새 서비스 추가 | PO 코드 수정 필요 | URL 등록만으로 완료 |
| 파라미터 파악 | 문서 읽고 수동 통합 | tools/list로 자동 수집 |
| LLM 연동 | function calling 코드 직접 작성 | tools/list 결과를 바로 LLM에 전달 |
| 서비스 상태 확인 | 별도 모니터링 구성 | health check 표준화 |
| 통신 규약 | 서비스마다 상이 | 항상 POST /mcp, JSON-RPC 2.0 |

---

## 우리 구조에서의 흐름

```
[에이전트 / LLM]
      ↓  tools/call pdf_to_jpg
[PO 오케스트레이터]  ← 어느 서버에 있는 툴인지 자동 라우팅
      ↓  tools/call pdf_to_jpg
[연구소 MCP 서버]  ← 내부적으로 기존 REST API 호출
      ↓
[실제 서비스 실행 결과 반환]
```

연구소 입장에서는 기존 서비스를 건드리지 않고  
MCP 어댑터(얇은 래퍼)만 씌우면 된다.

---

## 결론

MCP는 "새로운 통신 방식"이 아니라 **AI 에이전트가 외부 서비스를 쓰기 위한 표준 인터페이스**다.  
Anthropic이 MCP를 만든 이유가 정확히 이 상황 — LLM이 다양한 서비스를 연동할 때  
매번 커스텀 코드를 짜는 비효율을 없애기 위해서다.
