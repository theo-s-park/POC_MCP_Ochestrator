# Phase 4 - 멀티 서비스 MCP 연동 및 인증 전파

## 서비스 역할 정의

| 서비스 | 역할 | 인증 방식 |
|--------|------|-----------|
| **A** (기존 서비스) | 사용자 로그인, 세션 관리 | SID/CID 쿠키 |
| **B** (엔진) | 핵심 기능 처리 MCP 서버 | JWT 검증 |
| **C** (크레딧) | 기능 사용량 차감 MCP 서버 | JWT 검증 |
| **D** (오케스트레이터) | MCP 클라이언트 + 인증 교환 허브 | 쿠키 수신 → JWT 발급 |

---

## 전체 흐름

```
① 브라우저가 D에 요청 (SID/CID 쿠키 포함)
       ↓
② D → A: SID/CID 세션 검증 요청
       ↓
③ A → D: 유저 정보 반환 (userId, permissions 등)
       ↓
④ D: 내부 서비스용 JWT 발급
       ↓
⑤ D → B MCP: JWT 담아 엔진 tool 호출
       ↓
⑥ B: JWT 검증 → 엔진 기능 실행
       ↓
⑦ B → C MCP: JWT 담아 크레딧 차감 요청
       ↓
⑧ C: JWT 검증 → 크레딧 차감
       ↓
⑨ D → 브라우저: 최종 응답 반환
```

---

## 최종 검증 항목

### 1. 인증 교환 검증
- SID/CID 쿠키가 유효할 때 JWT가 정상 발급되는가
- SID/CID 쿠키가 만료/위조됐을 때 D에서 차단되는가
- 발급된 JWT에 필요한 클레임(userId, permissions)이 올바르게 담기는가

### 2. 인증 전파 검증
- D → B 호출 시 JWT가 Authorization 헤더로 정상 전달되는가
- B → C 호출 시 동일 JWT가 그대로 전파되는가
- JWT가 없거나 위조된 경우 B, C 각각에서 차단되는가

### 3. 서비스 연쇄 흐름 검증
- D가 B의 엔진 tool을 자연어로 호출할 수 있는가
- B의 tool 호출이 C의 크레딧 차감을 정상적으로 트리거하는가
- B 성공 + C 실패 시 어떻게 처리할 것인가 (정책 결정 필요)

### 4. 기존 서비스 호환성 검증
- A의 기존 쿠키 기반 브라우저 흐름이 영향 없이 동작하는가
- B, C가 쿠키 인증과 JWT 인증을 병행 처리할 수 있는가

---

## 구현 순서

```
Step 1. D에 세션 검증 + JWT 발급 레이어 구현
        └─ A의 /api/session/validate 호출
        └─ 검증 성공 시 내부 JWT 발급 (공유 시크릿 서명)

Step 2. B에 JWT 검증 미들웨어 추가
        └─ Authorization 헤더 파싱
        └─ 기존 쿠키 인증과 병행

Step 3. C에 JWT 검증 + 크레딧 차감 MCP 서버 구현
        └─ B가 C를 MCP Client로 호출하는 구조

Step 4. End-to-End 시나리오 검증
        └─ 자연어 요청 → D → B → C 전체 흐름 1회 통과
```

---

## 완료 기준

> 브라우저에서 SID/CID 쿠키로 D에 자연어 요청을 보냈을 때, D가 A에서 세션을 검증하고 JWT를 발급하여 B의 엔진 기능을 실행하고, B가 C의 크레딧을 차감하는 전체 흐름이 인증 단절 없이 end-to-end로 동작함을 확인한다. 또한 인증 실패 시 각 서비스에서 적절히 차단됨을 확인한다.
