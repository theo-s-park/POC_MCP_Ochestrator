# PO MCP 워크플로우 구조 협의 (2026-05-14)

## 현재 구현 상태 (Theo)

Phase 3 오케스트레이터 구축 완료. 아래 흐름이 실제로 동작 확인됨.

```
[에이전트 / LLM]
      ↓  POST /mcp  (tools/call)
[PO 오케스트레이터 :8080]
      ↓  툴이 어느 서버에 있는지 자동 라우팅
[MCP 서버 (연구소 서비스)]
      ↓  실제 서비스 실행
      ↑  결과 반환
```

- MCP 서버 등록/삭제 API 구현 완료
- 헬스체크 폴링 (60초, 3회 실패 시 INACTIVE) 구현 완료
- tools/list 자동 수집 구현 완료
- 외부 공개 MCP 서버 (`@modelcontextprotocol/server-everything`) 연동 검증 완료

## 1. 인증 방식

### 현재 (임시)
```http
POST /mcp
X-Mcp-Token: {임의 토큰}
```
PAT 포맷이 확정되지 않아 헤더 기반 임의 토큰으로 구현 중.

### 제안 방향
```http
POST /mcp
Authorization: Bearer {PAT}
```
PAT 검증 로직은 Woodie님 구현 방향에 맞춰 오케스트레이터 필터에 적용 예정.

- 1차: WAS 필터에서 PAT 검증 → 크레딧 차감 API 호출
- 2차: API Gateway로 이전

---

## 2. 서비스 상태 확인

**통신규약으로 확정 제안**: 모든 MCP 서버는 `GET /health` 필수 구현.

```http
GET {mcp-server-url}/health

Response 200 OK
{"status": "ok"}
```

오케스트레이터가 주기적으로 폴링하여 서버 상태를 `ACTIVE` / `INACTIVE`로 관리.

| 조건 | 처리 |
|------|------|
| 200 OK | ACTIVE 유지 |
| 연속 3회 실패 | INACTIVE 전환 |
| 복구 시 | 자동 ACTIVE 전환 |

---

## 3. 크레딧 차감 시점 제안

오케스트레이터가 `tools/call` 요청을 받아 MCP 서버로 라우팅하는 중간 지점에 위치하므로, **오케스트레이터에서 차감을 처리하는 것이 자연스럽다.**

### 제안: tools/call 성공 응답 시 차감

```
[에이전트]
   ↓ tools/call 요청
[오케스트레이터]
   ↓ MCP 서버로 포워딩
[MCP 서버]
   ↑ 실행 결과 반환
[오케스트레이터]
   → 크레딧 차감 API 호출  ← 이 시점
   ↑ 에이전트에게 결과 반환
```

**장점**: 실제 실행이 완료된 것만 차감 (요청만 하고 실패 시 미차감)
**협의 필요**: 차감 API 스펙, userId 식별 방식

### 미결 사항
- [ ] 차감 단위: 툴 호출 1회당 고정값? 서비스별 다른 값?
- [ ] userId는 PAT에서 추출? 별도 헤더?
- [ ] 크레딧 부족 시 오케스트레이터가 요청 차단 여부

## 5. 연구소 MCP 서버 구현 요건 (통신규약)

연구소가 기존 서비스에 MCP를 씌울 때 지켜야 할 최소 요건:

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `POST /mcp` (JSON-RPC 2.0) |
| 헬스체크 | `GET /health` → `{"status":"ok"}` |
| 프로토콜 버전 | `2024-11-05` |
| 전송 방식 | Streamable HTTP |

기존 REST API 건드릴 필요 없음. MCP 어댑터만 추가하면 됨.
